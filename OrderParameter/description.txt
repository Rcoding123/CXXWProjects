**Week 1 Project: Order Parameter Validator**

You are building a command-line tool that validates trading order parameters before they would be sent to an exchange. No trading logic, no market data — pure C++ foundations.

---

**What it does:**

The program accepts order details from the user via `std::cin` and validates them. It prints either a confirmation or a specific error message.

---

**Fields to collect and validate:**

- `symbol` — a string, 1 to 6 characters, uppercase letters only (e.g. `AAPL`, `ES`)
- `side` — must be exactly `BUY` or `SELL`
- `quantity` — a positive integer, greater than 0, no more than 10,000
- `price` — a positive decimal, greater than 0.0, no more than 999999.99
- `order_type` — must be exactly `LIMIT` or `MARKET`

---

**Rules:**

- If `order_type` is `MARKET`, price must be 0.0 (market orders have no price)
- If `order_type` is `LIMIT`, price must be greater than 0.0
- All fields must be present and valid before the order is accepted
- Print a specific error message for each validation failure — not just "invalid input"
- If all fields pass, print a formatted order summary

---

**Example output (valid order):**

```
=== Order Validator ===
Symbol: ES
Side: BUY
Quantity: 10
Price: 4500.25
Type: LIMIT

Order accepted.
--- Summary ---
BUY 10 ES @ 4500.25 [LIMIT]
```

**Example output (invalid):**
```
=== Order Validator ===
Symbol: es
ERROR: Symbol must be uppercase letters only (1-6 chars).
```

**What doesn't pass:**

**1. No `constexpr` for limits.**

The spec required it. You have magic numbers scattered through the code:

```cpp
std::cin.ignore(10000, '\n');  // magic number
if (quantity > 10000)          // magic number
if (price > 999999.99)         // magic number
```

Should be at the top of the file:

```cpp
constexpr int MAX_QUANTITY = 10000;
constexpr double MAX_PRICE = 999999.99;
constexpr int MAX_SYMBOL_LENGTH = 6;
constexpr int STREAM_IGNORE_LIMIT = 10000;
```

This is not style preference. In HFT code, magic numbers are a maintenance hazard. If max quantity changes, you want to change it in one place.

**2. `getPrice()` rejects price of 0.0 but spec allows MARKET orders to pass 0.0.**

You handled this correctly in `main` by not calling `getPrice()` for MARKET orders, but `getPrice()` itself enforces `price < 1.0` which would reject 0.0 anyway. Not a bug given your flow, but the validation boundary is slightly off — the spec says price > 0.0 for LIMIT, not price >= 1.0. A LIMIT order at $0.50 would be incorrectly rejected.

Fix:

```cpp
if (price <= 0.0) {
  std::cout << "Price must be greater than $0.00.\n";
  continue;
}
```

**3. `getOrderType()` prompt says `MARKET Or LIMIT` — inconsistent casing.** Minor but real. In production code prompts are part of the interface contract.

**4. `price` in `main` is default-initialized to 0.0 for MARKET orders implicitly.**

This works but it's implicit. A comment here would make intent clear:

```cpp
double price{0.0}; // 0.0 for MARKET orders, set by getPrice() for LIMIT
```
